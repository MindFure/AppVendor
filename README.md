# AppVendor: High Availability E-Commerce Infrastructure

Этот репозиторий содержит исходный код интернет-магазина автозапчастей и полностью автоматизированную, отказоустойчивую облачную инфраструктуру для его развертывания. Проект построен на принципах **Infrastructure as Code (IaC)** и объединяет в себе практики сквозной автоматизации — от создания виртуальных машин в облаке до настройки мониторинга и умного CI/CD-деплоя.

## 🚀 Архитектура и Стек технологий

Инфраструктура спроектирована по принципу обеспечения высокой доступности (**High Availability**) и исключает наличие единой точки отказа (SPOF) на всех критических уровнях.

* **Cloud Platform (IaC):** Cloud.ru (Evolution), Terraform
* **Frontend & Backend:** Nginx, PHP-FPM (контейнеризация на базе Docker)
* **High Availability Database Cluster:** PostgreSQL (Zalando Spilo), Patroni, etcd (распределенное хранилище состояний кластера СУБД)
* **Load Balancing & Routing:** HAProxy (Layer 4 TCP балансировка с Health Checks лидера Patroni)
* **IaC & Configuration Management:** Ansible, Python, Cron
* **CI/CD (Continuous Integration & Delivery):** Jenkins (Dynamic Pipelines)
* **Monitoring & Alerting:** Prometheus, Grafana, Node Exporter, Alertmanager (интеграция с Telegram Bot API через прокси)

## 📂 Структура репозитория

Репозиторий организован в соответствии с лучшими практиками разделения логики приложения, декларативного описания ресурсов и конфигураций автоматизации:

```text
AppVendor/
├── auto-parts-shop/      # Исходный код интернет-магазина (PHP/HTML)
├── configs/              # Изолированные конфигурации сервисов
│   ├── haproxy/          # Настройки балансировщика и панели метрик
│   ├── monitoring/       # Правила Prometheus и кастомные шаблоны Alertmanager
│   └── postgres/         # Скрипты инициализации (01_init.sql, 02_dump.sql)
├── deploy/               # Сердце автоматизации IaC
│   ├── ansible/          # Плейбуки, роли и Jinja2-шаблоны для деплоя приложений
│   └── terraform/        # Конфигурация манифестов Cloud.ru и скрипт динамического инвентаря inventory.sh
├── docker/               # Dockerfile для сборки кастомных окружений (Nginx, PHP)
├── scripts/              # Системные утилиты (сетевой Watchdog, бэкапы)
├── docker-compose.yml    # Главный декларативный манифест микросервисного кластера
└── Jenkinsfile.cloud     # Пайплайн автоматической сборки, создания ВМ и деплоя в облако
```

## 🛠 Ключевые особенности инфраструктуры

1. **Полный жизненный цикл инфраструктуры (IaC):** С помощью Terraform разворачиваются диски, сетевые порты и виртуальные машины на платформе Cloud.ru Evolution. Состояние инфраструктуры полностью прозрачно и контролируемо.
2. **Отказоустойчивая СУБД с автоматическим Failover:** Кластер баз данных управляется Patroni. При выходе из строя текущего Лидера, etcd мгновенно фиксирует событие, выбирает нового Лидера из Реплик, а HAProxy перенаправляет клиентские SQL-запросы на выживший узел. Время простоя стремится к нулю.
3. **Интеллектуальная инициализация БД:** При первичном развертывании Ansible автоматически определяет топологию кластера Patroni и безопасно накатывает файлы инициализации структуры и дампы товаров (`01_init.sql`, `02_dump.sql`) через балансировщик HAProxy. При повторных деплоях данные защищены от перезаписи.
4. **Динамический CI/CD Пайплайн:** Автоматизация в Jenkins (`Jenkinsfile.cloud`) самостоятельно создает облачные ресурсы, генерирует динамический инвентарь (`inventory.sh`) на основе актуального внешнего IP-адреса и запускает Ansible-плейбуки для идемпотентной настройки.
5. **Продвинутый умный алертинг (Telegram):** Настроен сбор метрик с помощью Prometheus и Node Exporter. Alertmanager отправляет уведомления в Telegram. Кастомные шаблоны на базе Go-templates разделяют логику уведомлений:
   * При падении сервисов приходит детальный аларм со статусом **🚨 АВАРИЯ**.
   * При восстановлении приходит лаконичное сообщение со статусом **✅ ВОССТАНОВЛЕНИЕ**, исключая фантомный спам старыми текстами ошибок во время плановых релизов.
   * Маршрутизация трафика к Telegram Bot API инкапсулирована через защищенный HTTP-прокси.

## ⚙️ Установка и запуск

### Требования
* Установленные `Ansible`, `Terraform` и `git` на управляющей ноде (или сервере Jenkins).
* Доступ к API Cloud.ru (сервисный аккаунт, access и secret ключи).
* Настроенный SSH-ключ для доступа к целевым инстансам.

### Вариант 1: Автоматический деплой через Jenkins (Облачная среда)
1. Создайте в Jenkins пайплайн, указав путь к репозиторию и `Jenkinsfile.cloud`.
2. Добавьте учетные данные облака (`TF_VAR_access_key`, `TF_VAR_secret_key`) в Masked Credentials.
3. Запустите **Build Now**. Конвейер автоматически создаст сервер в Cloud.ru, дождется доступности SSH, сформирует инвентарь, установит Docker-окружение и запустит магазин.

### Вариант 2: Локальный ручной запуск
1. Клонируйте репозиторий:
   ```bash
   git clone [https://github.com/MindFure/AppVendor.git](https://github.com/MindFure/AppVendor.git)
   cd AppVendor
   ```
2. Настройте секреты приложения в корневом файле `.env` (исключен из систем контроля версий):
   ```env
   DB_USER=myuser
   DB_PASSWORD=mypass
   DB_NAME=auto_part_shop
   ```
3. Запустите Ansible-плейбук для автоматической конфигурации ОС и запуска контейнеров:
   ```bash
   ansible-playbook deploy/ansible/playbook.yml -i deploy/ansible/inventory/hosts.ini -K
   ```
